## Scope and priorities

Release rehearsal and publication are explicitly **out of this batch**. S3
remains deferred until the language/runtime direction is more mature.

This batch starts with ownership contracts, then tackles the three remaining
lambda capture gaps, and only afterward expands collections. `Array<Interface>`
is a representation spike, not a promise to ship it in C20.

## Risks and mitigations

| Risk                                                   | Impact | Mitigation                                                                            |
| ------------------------------------------------------ | ------ | ------------------------------------------------------------------------------------- |
| Captured mutable values introduce GC/root bugs         | High   | Implement one capture kind at a time with stress corpus and root audits.              |
| Array views cannot be made safe without borrow types   | High   | Keep snapshots/clone as fallback and document rejection.                              |
| Iterator API forces a language-wide ownership change   | Medium | Design C20f first; do not implement mutation-through-entry without a stable contract. |
| `Array<Interface>` conflicts with the C backend layout | Medium | Keep C20h bounded and allow a defer outcome.                                          |

## C21 follow-up batch: ownership and developer experience

The following ten tasks are queued after C20j. C21a freezes borrow/ref as the
primary branch because it unlocks safe field returns and live collection views.
Async/tasks is deferred and must not be mixed into this C21 implementation
batch.

### Task C21a: Select and freeze the post-C20 direction

**Description:** Convert the C20j decision into one scoped implementation
contract and update the relevant RFCs.

**Acceptance criteria:**

- [x] Borrow/ref is selected as the primary C21 track because it unlocks safe Array field returns and collection views.
- [x] The MVP boundary explicitly lists syntax, runtime impact, and non-goals.
- [x] Async/tasks is recorded as deferred rather than mixed into C21.

**Verification:** RFC review with at least two positive examples (a local
borrow and a read-only field/view use) and two rejected examples (returning a
borrow and storing one in a closure/heap object).

**Dependencies:** C20j

**Files likely touched:** `docs/rfc/RFC-002-type-system.md`, `docs/rfc/RFC-003-memory-model-concurrency.md`, `docs/roadmap.md`.

**Estimated scope:** Medium

**Status:** Done. C21 is frozen around a scoped, non-owning `ref` capability:
`ref T` is ephemeral, non-null, lexical-scope-bounded, and cannot be stored or
escape through a return, closure, or task. The MVP is single-threaded and adds
no runtime ownership ABI; codegen may use a temporary pointer/view after sema
validation. Mutable borrows, nullable/nested refs, heap storage, live entry
mutation, and async/tasks are non-goals for this slice.

## C21 deferred work

The completed batch does not ship mutable borrows, nullable/nested/heap-stored
refs, ref escape through closures/tasks, live iterator or entry-view aliases,
mutation-through-entry, `Array<Interface>`, async/tasks/channels, concurrent
GC, or production release rehearsal/publication. These remain separate future
tracks and require new design/implementation plans; release is not requested.

## C21 direction decision

**Selected:** borrow/ref, implemented first as a scoped non-owning `ref T` MVP.
This directly addresses C20's remaining Array ownership debt and is a
prerequisite for safe Array field returns, live collection views, and any
future mutation-through-entry contract.

**Deferred:** async/tasks/channels, scheduler work, concurrent GC, and race
detection. They remain a separate future track and must not be mixed into the
C21 borrow implementation until the single-threaded lifetime contract is
validated.

## Agent allocation and conflict policy

Do not run the following implementation tasks as independent parallel edits
against the same checkout. Each worker owns a disjoint write set and returns a
small commit; integration happens in dependency order.

### Lane A: compiler/sema capture contract

- Scope: C20b/C20c sema metadata and diagnostics only.
- Write set: `crates/aura-sema/`, capture-related type-system docs, and focused diagnostic tests.
- Commit: `feat(sema): model mutable closure captures`
- Must not touch: `crates/aura-codegen/`, `runtime/aura_rt.c`, stdlib, or broad docs sync.

### Lane B: runtime ownership ABI

- Scope: C20c–C20e runtime retain/release and GC-root primitives.
- Write set: `runtime/aura_rt.c` and runtime unit/regression tests only.
- Commit: `feat(runtime): add capture ownership primitives`
- Can run alongside Lane A after the contract is frozen.

### Lane C: codegen lowering

- Scope: C20c–C20e C environment fields, load/store, and retain/release lowering.
- Write set: `crates/aura-codegen/` capture lowering files only.
- Commit: `feat(codegen): lower mutable closure captures`
- Depends on Lane A and Lane B; must not modify runtime or sema contracts.

### Lane D: corpus/integration coverage

- Scope: C20c–C20e positive, negative, and stress programs.
- Write set: `corpus/fun/`, narrowly scoped `corpus/class/`/`corpus/generic/`, and `corpus/README.md`.
- Commit: `test(corpus): cover mutable closure capture ownership`
- Starts after Lane C; no compiler/runtime edits.

### Lane E: stdlib and DX sidecars

- Collections snapshot tests, std.io Result contract, formatter, and diagnostic/report models are separate tasks.
- Only one worker may own `crates/aura-cli/src/main.rs` at a time.
- Keep CLI model/serialization work in new modules until a final integration commit.
- Recommended commits: `test(stdlib): lock collection snapshot semantics`, `docs(std.io): define non-throwing result contract`, `feat(cli): add aura fmt MVP`, and `feat(dx): add structured diagnostic and test report models`.

### Integration and commit order

```text
Contract docs/sema ─┐
Runtime ABI ────────┴─> Codegen ─> Corpus ─> Docs/status close
                         │
                         └─> DX/stdlib sidecars (parallel when write sets differ)
```

- Every agent must report changed files and run the narrowest relevant tests before committing.
- Do not use shared broad commits such as `feat: C20`; one logical lane per commit.
- The parent integrates/rebases commits in dependency order and runs `cargo test --workspace` after each integration checkpoint.
- Borrow/ref, live iterators, and `Result` runtime implementation stay out of the capture lane; they open only after C20 capture commits are merged.
