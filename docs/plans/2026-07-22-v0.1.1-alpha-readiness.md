# Aura v1 Readiness Implementation Plan

## Overview

This plan closes the gaps identified by the 2026-07-22 RFC audit before Aura is
called v1-ready. It covers the compiler backend, async lowering, attributes and
macros, race detection, build profiles/cache/targets, and the CLI/documentation
contract.

The current repository is a working C-backend compiler MVP. The v1 target is a
reproducible Rust toolchain that produces a stable native executable, lowers the
accepted async model safely, exposes the accepted attribute/macro surface,
detects development-time races, supports profile/cache/target build behavior,
and documents only commands and guarantees that actually ship.

This is an implementation plan, not a proposal to rewrite the accepted RFCs.
Where an RFC leaves a choice open, this plan chooses the smallest implementation
that satisfies the existing contract and records the choice below.

## Baseline and evidence

- Compiler crates currently include AST, lexer, parser, sema, C codegen,
  diagnostics, and CLI; there are no HIR/MIR/LLVM crates yet.
- The active codegen path emits C and invokes the host `cc` compiler from
  `crates/aura-codegen/src/build.rs`.
- Async syntax, task frames, deterministic FIFO execution, and bounded channels
  exist, but await suspension, non-empty spawn captures, async frame GC roots,
  and complete task failure propagation remain incomplete.
- The shipped stdlib is `std/io`, `std/assert`, and `std/collections`; network,
  JSON, logging, crypto, synchronization, and async I/O are not v1 readiness
  blockers unless a later product decision promotes them.
- The CLI currently supports `new`, `init`, `version`, `check`, `build`, `run`,
  `test`, `fmt`, and `emit-c`.
- The current restricted environment produced 74/79 passing workspace tests;
  the five failures are network/cache-permission-sensitive and must be rerun on
  a normal host before using the result as a release signal.

## Definition of v1 readiness

Aura may be labeled v1-ready only when all of the following are true:

- [ ] One selected native backend is the default production backend and has a
      documented support matrix. The C backend is either retained only as a
      test/debug backend or explicitly promoted with a stable portability
      contract.
- [ ] `async`/`await`/`spawn`/`join`/`cancel` and bounded channels have executable
      semantics for non-empty bodies, suspension, captures, cancellation, and
      success/failure outcomes.
- [ ] Accepted built-in attributes, derives, and declarative macros work through
      parse, expansion, sema, codegen, diagnostics, and package boundaries.
- [ ] Development builds can detect a planted data race, while release builds
      do not enable the detector accidentally.
- [ ] `dev`, `test`, and `release` profiles, deterministic artifact cache,
      target selection, and single-executable linking work end-to-end.
- [ ] CLI help, exit codes, flags, RFCs, roadmap, release scripts, and examples
      agree with the actual implementation.
- [ ] A clean-host acceptance matrix passes for the supported targets and the
      repository has no unresolved v1 blocker in `agents/debts.md`.

## Architecture decisions

1. **Backend direction:** implement the RFC-004 backend abstraction first and
   make LLVM the production backend, because LLVM is the accepted v1 direction.
   Keep C codegen behind the same driver as a compatibility/debug backend until
   LLVM reaches corpus parity; do not delete the C backend during the migration.
2. **IR boundary:** introduce typed HIR and CFG-based MIR before LLVM lowering.
   Async state machines, macro expansion metadata, debug spans, and future LSP
   queries all need a stable compiler-internal boundary.
3. **Runtime ABI:** version the task, GC-root, exception, channel, and FFI ABI.
   A task frame must own or explicitly root every value that survives a poll;
   borrowed values cannot cross an await or task boundary.
4. **Macro safety:** ship built-in derives and hygienic declarative macros first.
   Procedural macros remain a separate, sandboxed phase and are not a v1 gate.
5. **Race detection:** use an instrumented development runtime/compiler mode
   with a deterministic report format. Release mode remains free of detector
   overhead unless explicitly requested.
6. **Build reproducibility:** key artifacts by compiler version, backend,
   target, profile, feature set, source hash, dependency lock hash, and runtime
   ABI version. A cache miss must be safe and deterministic; a corrupt cache
   entry must be discarded and rebuilt.

## Dependency graph

```text
Contract freeze and compatibility corpus
        │
        ├── Backend/IR abstraction ──► LLVM backend ──► target/profile linking
        │                                      │
        │                                      └── async lowering + runtime ABI
        │                                                 │
        │                                                 └── race detector
        │
        ├── Attribute/metadata model ──► derives ──► declarative macros
        │                                      │
        │                                      └── test discovery integration
        │
        └── CLI/build/documentation contract ──► release acceptance matrix
```

Independent work can begin on the contract corpus, attribute syntax design,
and build-cache schema after Task 0. LLVM lowering, async lowering, and race
detection must share the backend/runtime ABI and should not be implemented as
uncoordinated parallel rewrites.

## Task list

### Phase 0 — Contract freeze and measurement

### Task 0.1: Establish the v1 contract matrix

**Description:** Create one machine-readable and human-readable matrix mapping
each accepted requirement to its implementation owner, test corpus, CLI surface,
and status. Resolve the remaining C-backend-versus-LLVM decision, supported
target matrix, macro scope, race-detector mode, and profile names before feature
work starts.

**Acceptance criteria:**

- [ ] Every v1 gate in this document maps to RFC sections and a repository test.
- [ ] “Implemented”, “partial”, “deferred”, and “out of scope” have explicit
      meanings and are used consistently across roadmap and RFC Toolchain today
      notes.
- [ ] The matrix identifies the exact supported target triples and the required
      host linker/sysroot for each.

**Verification:** Review the matrix against RFC-003, RFC-004, RFC-008,
RFC-009, RFC-010, RFC-011, RFC-012, RFC-013; run the existing corpus and CLI
smoke tests and record the baseline result.

**Dependencies:** None.

**Files likely touched:** `docs/plans/2026-07-22-v1-readiness.md`,
`docs/roadmap.md`, optionally `docs/v1-contract.md`.

**Estimated scope:** Medium.

### Task 0.2: Add a compatibility and golden-test harness

**Description:** Separate frontend/sema corpus tests, backend output tests,
runtime ABI tests, CLI contract tests, and release acceptance tests. Add golden
fixtures for diagnostics and generated IR/C/LLVM where output stability matters.

**Acceptance criteria:**

- [ ] A single command runs the v1 readiness suites with stage labels.
- [ ] Existing C-backend corpus remains a compatibility suite during migration.
- [ ] Expected failures for deferred features are explicit negative tests rather
      than undocumented behavior.

**Verification:** Run the harness on a clean checkout; confirm failures identify
the phase, target, and fixture rather than only returning a generic exit code.

**Dependencies:** Task 0.1.

**Files likely touched:** `scripts/`, `corpus/`, `crates/*/src/tests.rs`,
`runtime/tests/`, `Cargo.toml`.

**Estimated scope:** Medium.

### Checkpoint: Contract baseline

- [ ] Existing corpus passes on the current host.
- [ ] Network/cache-sensitive tests have a documented clean-host procedure.
- [ ] No implementation task starts while the backend, target, or macro scope is
      still ambiguous.

### Phase 1 — Compiler IR and production backend

### Task 1.1: Introduce backend and target driver abstractions

**Description:** Extract the current compile flow into a driver that accepts a
backend, target triple, profile, feature set, and runtime ABI version. Preserve
the current C path as `--backend c` or an internal compatibility backend while
making the API independent of C source generation.

**Acceptance criteria:**

- [ ] Frontend/sema runs once and produces a backend-neutral checked program.
- [ ] C backend uses the new driver without behavior changes.
- [ ] Unsupported backend/target combinations produce structured diagnostics.

**Verification:** Build and run hello, packages, generics, exceptions, lambdas,
GC, and tests through both the old and new driver paths; compare exit status and
structured diagnostics.

**Dependencies:** Tasks 0.1–0.2.

**Files likely touched:** `crates/aura-codegen/src/build.rs`,
`crates/aura-codegen/src/lib.rs`, `crates/aura-cli/src/main.rs`, new backend
driver modules.

**Estimated scope:** Large; split into backend interface and C adapter commits.

### Task 1.2: Add typed HIR and CFG-based MIR

**Description:** Lower the checked AST into typed HIR and then MIR with explicit
control-flow, calls, ownership/drop points, exception edges, GC roots, and mono
instantiations. Keep spans and source file IDs on every user-visible operation.

**Acceptance criteria:**

- [ ] Existing non-async corpus lowers to HIR/MIR deterministically.
- [ ] Generic monomorphization and interface dispatch are represented explicitly.
- [ ] Return, throw, catch, cleanup, and GC-root behavior are visible in MIR.
- [ ] MIR dumps are available for compiler tests, even if not yet stable CLI API.

**Verification:** Add MIR golden tests for class calls, generic arrays, interface
dispatch, nullable flow, exceptions, lambda captures, and GC cleanup.

**Dependencies:** Task 1.1.

**Files likely touched:** new `crates/aura-hir/`, `crates/aura-mir/`,
`crates/aura-sema/`, `crates/aura-ast/`, `Cargo.toml`.

**Estimated scope:** Large; break into HIR, MIR CFG, and cleanup/mono slices.

### Task 1.3: Implement LLVM codegen to corpus parity

**Description:** Add LLVM lowering for the v1 MIR subset, including native calls,
classes/structs/enums/interfaces, generics, exceptions, GC safepoints/root
registration, strings/arrays, lambdas, and debug spans. Link the runtime through
an explicit ABI rather than embedding ad hoc C declarations in emitted source.

**Acceptance criteria:**

- [ ] LLVM backend compiles every currently passing positive corpus case on at
      least one host target.
- [ ] Negative diagnostics remain produced by the shared frontend/sema path.
- [ ] Generated binaries pass runtime, memory-safety, and test-runner suites.
- [ ] C backend remains available only as compatibility/debug until parity is
      demonstrated for all v1-supported constructs.

**Verification:** Compare C and LLVM results on golden programs; run sanitizer
and GC churn tests; inspect LLVM IR for roots, cleanup, and exception edges.

**Dependencies:** Tasks 1.1–1.2.

**Files likely touched:** new `crates/aura-llvm/`, `crates/aura-codegen/`,
`runtime/`, `Cargo.toml`, CI workflow files.

**Estimated scope:** XL; split by language/runtime feature families.

### Task 1.4: Define and test the stable runtime ABI

**Description:** Version the ABI shared by compiler and runtime. Specify task
frames, result ownership, exception payloads, GC root registration, channel
payload destruction, cancellation, and FFI boundaries. Add ABI compatibility
checks so compiler/runtime mismatches fail early.

**Acceptance criteria:**

- [ ] ABI version is embedded in compiler output and runtime metadata.
- [ ] Every owned pointer crossing a frame/channel/FFI boundary has a matching
      destroy or transfer rule.
- [ ] Borrowed references are rejected across await, spawn, channel send, and
      returned task state.

**Verification:** Compile ABI fixtures against the runtime; run ASan/UBSan and
GC stress tests; intentionally use an incompatible ABI and verify a clear error.

**Dependencies:** Tasks 1.1–1.2; coordinates with Task 2.1.

**Files likely touched:** `runtime/aura_rt.c`, `runtime/tests/`,
`crates/aura-codegen/`, `crates/aura-llvm/`, `crates/aura-sema/`.

**Estimated scope:** Large.

### Checkpoint: Production backend

- [ ] LLVM and C produce equivalent results for the compatibility corpus.
- [ ] Runtime ABI tests pass under sanitizers.
- [ ] At least one clean-host target can build and run a single executable with
      no runtime installation.

### Phase 2 — Async lowering and runtime completion

### Task 2.1: Lower async functions into MIR state machines

**Description:** Represent each await point as a state transition with explicit
locals, captures, result storage, cancellation checks, and resume locations.
Reject unsupported borrowed values before lowering rather than relying on C or
LLVM behavior.

**Acceptance criteria:**

- [ ] Async functions with one and multiple await points suspend and resume.
- [ ] Locals and owned values survive suspension with correct drop behavior.
- [ ] Exceptions and cancellation become defined task outcomes.
- [ ] Empty no-await async remains supported through the same state-machine path.

**Verification:** Add deterministic tests for immediate completion, delayed
completion, multiple awaits, cancellation, thrown exceptions, string/array/class
captures, and borrow rejection at await.

**Dependencies:** Tasks 1.2 and 1.4.

**Files likely touched:** `crates/aura-mir/`, `crates/aura-codegen/`,
`crates/aura-llvm/`, `crates/aura-sema/`, `runtime/aura_rt.c`.

**Estimated scope:** Large.

### Task 2.2: Implement non-empty spawn captures and task outcomes

**Description:** Lower `spawn` bodies into task frames with owned capture
storage, explicit mark/drop hooks, join handles, cancellation state, and failure
propagation. Define whether unjoined failures are logged, stored, or surfaced
according to RFC-003.

**Acceptance criteria:**

- [ ] Non-empty spawn bodies execute exactly once and can yield at await points.
- [ ] Captured Int/String/class/Array/Fun values remain valid until task finish.
- [ ] Join observes success, failure, and cancellation deterministically.
- [ ] Unjoined task failure follows the documented supervision policy.

**Verification:** Add task lifecycle/churn tests and end-to-end Aura corpus cases;
run ASan/UBSan and forced-GC tests while tasks are suspended.

**Dependencies:** Task 2.1.

**Files likely touched:** `runtime/aura_rt.c`, `runtime/tests/`,
`crates/aura-codegen/`, `crates/aura-llvm/`, `corpus/async/`.

**Estimated scope:** Large.

### Task 2.3: Complete channel and async I/O boundaries

**Description:** Keep bounded channels deterministic for the single-threaded
MVP, but make send/receive suspension and payload ownership work with the full
task state machine. Add only the async I/O surface committed as v1; do not add an
HTTP framework.

**Acceptance criteria:**

- [ ] Full/empty channel operations suspend and resume without leaks or lost
      messages.
- [ ] Typed class/String/Array payload cleanup is correct on close/cancel/failure.
- [ ] Async I/O APIs either work end-to-end or are explicitly removed from the
      v1 contract and RFC Toolchain today notes.

**Verification:** Deterministic producer/consumer tests, close/cancel tests,
GC-under-backpressure tests, and one CLI-oriented async I/O example.

**Dependencies:** Task 2.2.

**Files likely touched:** `runtime/aura_rt.c`, `runtime/tests/`, `std/`,
`crates/aura-codegen/`, `corpus/async/`.

**Estimated scope:** Large.

### Checkpoint: Async correctness

- [ ] No-await and suspension paths use one consistent frame ABI.
- [ ] All task/frame/channel tests pass under ASan/UBSan.
- [ ] No async debt remains that contradicts the v1 contract.

### Phase 3 — Attributes, metadata, derives, and declarative macros

**Description:** Add AST representation, declaration validation, known-attribute

**Acceptance criteria:**

- [ ] Attributes parse on all RFC-009-supported declaration sites.
- [ ] Unknown attributes produce a stable diagnostic and error code.
- [ ] Metadata can be retained for compile-time tooling and optionally emitted
      into a binary without changing ordinary codegen semantics.
- [ ] `@test` is migrated to the same attribute infrastructure.

**Verification:** Parser/sema negative tests, metadata golden files, and test
discovery regression tests.

**Dependencies:** Task 1.2.

**Files likely touched:** `crates/aura-lexer/`, `crates/aura-ast/`,
`crates/aura-parser/`, `crates/aura-sema/`, `crates/aura-diagnostics/`,
`crates/aura-codegen/`.

**Estimated scope:** Large.

### Task 3.2: Implement built-in derives

**Description:** Add the RFC-010 MVP derive pipeline for the smallest useful

IR, preserving hygiene and source attribution.

**Acceptance criteria:**

- [ ] `@derive(Equals)`, `@derive(Hash)`, and `@derive(Debug)` work on the
      supported class/struct/enum subset.
- [ ] Generated members participate in normal sema, generics, visibility, and
      codegen checks.
- [ ] Derive errors point to the attribute and the invalid field/type.

**Verification:** Positive/negative corpus for primitive, nested, generic,

**Dependencies:** Task 3.1.

**Files likely touched:** new `crates/aura-derive/` or macro module,

**Estimated scope:** Large.

### Task 3.3: Implement hygienic declarative macros

**Description:** Implement the accepted pattern-macro syntax and expansion phase

limits, package export/import behavior, and diagnostics for unmatched patterns.

**Acceptance criteria:**

- [ ] Local and imported macros expand before name resolution in a deterministic
      order.
- [ ] Macro-introduced identifiers cannot accidentally capture caller locals.
- [ ] Expansion depth/token limits prevent runaway compilation.
- [ ] Macro errors include call-site and definition-site spans.

**Verification:** Hygiene, nested expansion, package boundary, ambiguity,
recursion-limit, and diagnostic golden tests.

**Dependencies:** Tasks 1.2 and 3.1.

**Files likely touched:** `crates/aura-lexer/`, `crates/aura-parser/`, new

### Checkpoint: Language extension safety

### Phase 4 — Race detector and concurrency tooling

### Task 4.1: Freeze the race-detector instrumentation model

**Description:** Specify the runtime/compiler events that form synchronization
and access edges: reads, writes, task spawn/join, await resume, channel
send/receive, locks, atomics, and cancellation. Define report identity,
stack/source mapping, suppression policy, and dev/release defaults.

**Acceptance criteria:**

- [ ] The model maps directly to RFC-003 happens-before rules.
- [ ] Instrumentation is disabled by default in release builds.
- [ ] Reports identify both conflicting accesses and their source spans.

**Verification:** Design review plus a planted-race fixture and a race-free
fixture whose reports are stable across repeated runs.

**Dependencies:** Task 2.2; can be designed in parallel with Phase 3.

**Files likely touched:** `docs/`, `runtime/`, `crates/aura-mir/`,
`crates/aura-diagnostics/`, `corpus/concurrency/`.

**Estimated scope:** Medium.

### Task 4.2: Implement instrumented runtime/compiler mode

**Description:** Add compiler instrumentation and runtime tracking for task and
shared-memory operations. Start with deterministic single-threaded scheduling,
then preserve the event model for future worker threads without claiming data
race safety that the runtime does not provide.

**Acceptance criteria:**

- [ ] A deliberate conflicting access is reported with two stacks/spans.
- [ ] Synchronization and channel edges suppress false positives correctly.
- [ ] `aura test --race` or the selected equivalent is documented and stable.
- [ ] Ordinary `aura build` and release binaries do not include detector state.

**Verification:** Race-positive, race-negative, channel ordering, cancellation,
GC, and performance smoke suites; compare instrumented/non-instrumented output.

**Dependencies:** Task 4.1 and the final task/frame ABI from Phase 2.

**Files likely touched:** `runtime/aura_rt.c`, `runtime/tests/`,
`crates/aura-mir/`, `crates/aura-llvm/`, `crates/aura-cli/src/main.rs`,
`crates/aura-diagnostics/`.

**Estimated scope:** Large.

## Recommended implementation order

1. Tasks 0.1–0.2: freeze contracts and make current behavior measurable.
2. Tasks 1.1–1.4: establish backend/IR/runtime ABI; keep C compatibility green.
3. Tasks 5.1–5.3: build profiles/cache/targets can proceed once the driver and
   ABI exist, but target claims must wait for backend evidence.
4. Tasks 2.1–2.3: complete async lowering and runtime ownership on the final IR.
5. Tasks 3.1–3.3: implement attributes/derives/macros after HIR metadata and
   diagnostics boundaries are stable.
6. Tasks 4.1–4.2: implement race detection after task/frame synchronization
   events are final.
7. Tasks 6.1–6.3: close CLI and documentation contracts against shipped code.
8. Tasks 7.1–7.3: hardening, release acceptance, and final decision.

## Parallelization opportunities

- Tasks 0.1–0.2 and the design portion of 4.1 can run in parallel.
- Task 3.1 attribute syntax and Task 5.1 profile schema can run in parallel once
  the contract matrix is approved.
- Runtime tests, negative corpus tests, and documentation consistency checks can
  run in parallel after each relevant interface is frozen.
- LLVM lowering, async lowering, and ABI implementation should remain coordinated
  because they share ownership, exception, GC-root, and debug-span semantics.

## Risks and mitigations

| Risk                                                             | Impact   | Mitigation                                                                                     |
| ---------------------------------------------------------------- | -------- | ---------------------------------------------------------------------------------------------- |
| LLVM migration changes ownership or exception behavior           | Critical | Keep C compatibility backend, add MIR/ABI golden tests, migrate feature families incrementally |
| Async frames retain invalid pointers across await                | Critical | Explicit frame mark/drop hooks, borrow barriers, sanitizer + forced-GC tests                   |
| Macro hygiene creates silent name capture                        | High     | Hygiene IDs, expansion limits, call/definition spans, adversarial corpus                       |
| Race detector reports false positives or misses channel edges    | High     | Freeze happens-before model first; deterministic positive/negative fixtures                    |
| Cache returns artifacts built with stale runtime/compiler inputs | High     | Content-addressed keys include ABI/compiler/target/profile/lock hashes                         |
| Cross-target build passes compilation but fails at runtime       | High     | Separate compile and native-runtime claims; require matching-host acceptance                   |
| CLI/RFC drift recurs                                             | Medium   | Generate status/help tables from one contract matrix and make docs a release gate              |
| Scope expands into full stdlib/framework work                    | Medium   | Keep networking/JSON/frameworks deferred unless explicitly promoted to v1                      |

## Open questions requiring product decision

- Is LLVM mandatory for the v1 release, or can a production-hardened native C
  backend satisfy v1 temporarily? This plan assumes LLVM is mandatory because
  RFC-004 names it as the v1 primary backend.
- Which exact target triples are v1-supported, and which are compile-only versus
  native-runtime-tested?
- Are `aura add`, `aura publish`, workspaces, and `aura completions` v1 gates or
  explicitly post-v1? The plan permits deferring them, but the CLI/RFC contract
  must say so consistently.
- Is async I/O part of v1, or is the v1 async contract limited to deterministic
  task/channel primitives?
- What is the policy for an unjoined failed task: log, process failure, or a
  retained outcome visible through a supervisor?
- What is the supported FFI surface for v1, and how are foreign pointers rooted,
  pinned, and released?
